# biz-to-leet

## 这个 Skill 是什么？

这是一个帮你把**工作中的真实业务问题**提炼为**可刻意练习的纯函数 LeetCode 风格题目**的 Skill。

它做的事情，类似于一个私人出题老师：当你遇到一个业务问题，它帮你剥离掉所有无关的工程噪音（数据库、HTTP 框架、日志、配置文件等），把核心逻辑提炼出来，定义清楚"给什么输入、返回什么输出、有什么约束"，形成一道风格类似 LeetCode 的纯函数题目——然后同步生成配套的单元测试，让你可以直接上手练习和验证。

## 为什么需要它？

### 陷阱一：AI 生成代码，工程师却失去了真正的成长

很多工程师虽然有丰富的工作经验、接触过复杂的业务逻辑，但日常工作中大量依赖 AI 生成代码，**真正需要自己动脑想清楚的核心逻辑，全都没有经过独立思考**，把自己淹没在AI生成代码的毫无成长意义的工作循环里，没有被刻意练习过。久而久之，技术能力悄悄退化，丧失职业竞争力。

### 陷阱二：知识碎片化，没有办法系统沉淀

在工作中学到的东西太零碎，每个问题带来的思考随任务结束就消散了，没有被保留下来。这个 Skill 为每个业务问题生成一道题，**每道题一个文件夹，按时间戳归档**，形成你自己的私人题库，积累越多，价值越大。

### 陷阱三：训练抽象建模思维

业务问题从来不会以"输入/输出"的形式呈现给你。遇到业务问题，往往思路混乱、不知从何下手——你收到的通常是上级说的一段话或一个需求文档，比如：

> "设备上架之后，要根据机房、机柜、用途自动生成设备名，格式按我发你的 Excel 来。"

**抽象建模强迫你**把这团模糊的描述，提炼成精确的结构：

```
输入：room="DC01", rack="A01", role="接入交换机", index=3
输出："DC01-A01-ACC-03"
约束：index 从 1 开始，不足两位补零
```

当你能写出这三行，思路也就自然理清了。

### 陷阱四：训练 Edge Case 防御性编程直觉

你的脑子天然会忽略"不正常情况"。比如你写完 IP 分配函数，测了几个正常 case 通过了，但你根本不会主动去想：

- `used` 里传进来一个重复 IP 怎么办？
- `subnet` 是 `"10.0.0.0/31"` 这种边缘掩码怎么办？
- `n` 传了个 `-1` 怎么办？

于是代码上线，某天某个边界情况触发了，出了 Bug，你才恍然大悟。

当训练了大量有 Edge Case 的题之后，你在写任何函数时，脑子里都会自动跳出来问自己：我还能想到什么奇怪的输入？空的怎么办？负数怎么办？重复的怎么办？**这种直觉，不是靠看书学来的，是靠大量刻意练习刻进去的。**

## 为什么要生成 "属于自己的 LeetCode" ？

LeetCode 的价值在于它剥离了所有工程细节，把一个问题压缩到最小的可练习单元。但 LeetCode 的缺点是**题目和工作脱节**，练了感觉没什么用。

而这个 Skill 要做的，就是从你自己的真实工作中，持续生产属于你自己的"真实业务场景版的生动 LeetCode"，**用刻意练习来逼迫自己成长**。

### 使用流程

每次使用时，按如下步骤操作：

1. 你遇到一个真实的业务问题（哪怕只有模糊的描述）
2. 告诉 AI：`"帮我把这个问题做成一道练习题"`
3. AI 输出：题目 `.md` + 空函数 `.go` + 测试 `_test.go`
4. 你自己实现函数，运行 `go test ./...` 验证
5. 题目持续积累到 `problems/` 目录，逐步形成你的私人高质量题库

### 单元测试要求

AI 在生成题目的同时，须为对应纯函数自动生成配套的单元测试文件，具体要求如下：

- 采用**表格驱动（Table-Driven）**的测试风格
- 测试用例**不少于 100 个**，须覆盖以下三类场景：
  - **正常路径（Happy Path）**：输入合法、逻辑正常的典型调用
  - **边界条件（Boundary Cases）**：如空输入、极值、极大/极小数据集等
  - **易出现 Bug 的场景（Edge Cases）**：如整数溢出、空指针、重复元素、乱序、特殊字符等

### 输出规范

- 每道题目输出到项目根目录 `problems/` 下，**每道题独占一个子目录**，目录命名格式为 `<YYYYMMDDHHmmss> <题目名>`（精确到秒的时间戳加题名，便于按时间排序归档）。

- 每个子目录内包含以下**三个文件**：

  ```
  problems/
    └── 20260303183000 IP地址分配/
        ├── assign-ips.md          # 题目描述（Markdown 格式，含输入输出说明、示例、约束）
        ├── assign_ips.go          # 纯函数签名 + panic("not implemented") 占位实现
        └── assign_ips_test.go     # 表格驱动单元测试（≥100 个用例）
  ```

- **`## 题目描述` 下必须包含两个子节**：

  ```markdown
  ## 题目描述
    ├── ### 业务场景   ← 必须按"问题背景 → 导致的问题 → 解决方案"三段式结构：
    │                    1. 问题背景：描述真实工作场景和传统做法
    │                    2. 导致的问题：列出现状的痛点（效率、准确性、可维护性等）
    │                    3. 解决方案：说明程序如何解决这些痛点，带来什么价值
    │                    让读者理解为什么要做，一眼认出"这正是工作中真实会碰到的问题"，并理解程序存在的必要性，产生做题的动力。
    └── ### 核心问题   ← 剥离业务噪音后，精准描述算法的本质（输入、输出、约束）
  ```

- **题目描述必须通俗易懂**，避免使用晦涩的专业术语。若必须使用专业术语（如"跳号"、"广播地址"），须在首次出现时用括号标注通俗解释。

- **单题原则**：每次对话只生成一道题目。若业务场景可拆解为多个子题，须由我主动声明，AI 不得擅自拆分。

- **相关知识点**：最后在"相关知识点"部分明确列举做这道题所需的前置知识，便于我提前有针对性地复习。

- **编程语言**：`Go`。

- **示例格式要求**：
  - 示例标题格式：`## 示例 N：示例的描述`（如"示例 1：基本分配"）
  - 输入输出优先使用 JSON 格式（带语法高亮）
  - 每个示例必须包含"解释"部分，说明为什么这个输入的到这个输出的关键逻辑
  - 至少提供 3 个示例：正常场景、边界条件、错误场景
