# 文档结构规范

## 一、设计理念（为什么用大纲）

- **工具类比** 文章用大纲笔记的风格写（类似 workflowy、logseq），一个意思的段落为一个 bullet。

- **反对散文** 一段话塞三四个论点、长句套从句、过渡词“此外 / 另外 / 与此同时”满天飞——这些一律避免。

- **示例对照**

    - **坏写法（散文式）**

        > 校验顺序是首先加载 license.json，然后用内嵌公钥验 signature，接着现场算本机指纹，还要检查 issued_at 和 expires_at，以及 T_last，最后检查账本长度。任何一步失败都要拒跑并统一报错。

    - **好写法（大纲式）**

        - **校验流程**
            1. 加载 `license.json`
            2. 验 `signature`
            3. 比对指纹
            4. 检查时间窗
            5. 检查 `T_last`
            6. 检查账本长度

        - **失败处理** 任何一步失败直接退出，统一报 `license 无效`。

## 二、整体骨架

- **`#` 一级标题** 整篇文档唯一一个，作为文档名。

- **`##` 二级章节** 用中文数字编号（如 `## 一、目标`、`## 二、xxx`），负责对 H3 分组。章节内容少时可以直接跟列表项，不强求套 H3。

- **`###` 三级章节** H3 下是大纲笔记正文。标题简洁，不重复上层 H2 已经交代的信息。

- **不使用 `####` 及以下** 所有细节通过列表缩进层级表达。

## 三、大纲笔记格式

- **列表项前置加粗关键词** 每个列表项以 `- **关键词**` 开头，关键词概括这一段的主题。读者应该能只扫加粗关键词就理解全文骨架，不需要读正文也能抓到主线。

- **关键词之后直接接正文** 关键词和内容之间不加冒号，用空格分隔，整句读起来自然。

- **列表项之间必须空一行** 视觉分离，便于阅读。

- **项目内部不再加粗** 加粗只用于列表项开头的关键词，正文里不再出现 `**xxx**`，避免视觉干扰（强调用双引号或代码反引号即可）。

## 四、嵌套层级

- **子点用 4 空格缩进** 父列表项下的子列表、代码块、表格、有序步骤都用 4 空格缩进归属到父项。

- **子点不必再加粗关键词** 子点是对父项的展开，保持简洁即可。

- **有序步骤用 `1. 2. 3.`** 表示时序 / 流程的用有序列表；无时序关系的用无序 `-` 列表。

- **嵌套子点的用途** 子点用于列举父项的具体构成——参数列表、步骤、示例；代码块 / 表格 / JSON 都作为父项的展开，4 空格缩进归属在父项下。

## 五、引号和标点

- **中文句子用中文双引号 `“”`** 不使用英文 `""`。

- **代码 / 命令 / 字段名用反引号** 如 `license.json`、`--force-reissue`、`pkg/license/`。

- **句末正常使用中文标点** 每个列表项以句号结尾，半句的子弹点可省略句号。
